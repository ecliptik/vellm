# Phase 3 Notes — Speed, KV compression, correctness-gate upgrade

Companion to `docs/phase1-notes.md` and `docs/phase2-memory.md`. This
document captures harness-side Phase 3 decisions and measurements:

- `--max-seq-len` CLI flag (task #1)
- tolerance-based correctness gate (task #2)
- final integration benchmark + report (task #5)

The matmul and int8-KV-cache work (tasks #3, #4) is `tuner`'s lane;
this document references those numbers where they intersect with the
harness work but does not own them.

## Task #1 — `--max-seq-len` CLI flag

`stories42M_q80.bin` advertises `seq_len=1024` in its header. The KV
cache is sized `2 × n_layers × seq_len × (dim × n_kv_heads / n_heads)
× sizeof(float)` which on 42M is `2 × 8 × 1024 × 512 × 4 = 32 MB` —
the single largest allocation post-Phase-2 and the reason 42M still
paged ~30 MB through `CWSDPMI.SWP` on the 48 MB target (see
`docs/phase2-memory.md` §"stories42M_q80 upper bound").

Most prompts + generations don't need 1024 tokens. Upstream `runq.c`
already silently caps `steps` at `config.seq_len`, but the cache is
allocated at `config.seq_len` regardless — the extra ~24 MB of ring
sits unused.

### Flag contract

- `--max-seq-len N` (long form) and `-L N` (short alias) both accept
  a positive integer.
- When set, caps `transformer.config.seq_len = min(config.seq_len, N)`
  **before** `runstate_arena_size()` is computed — the arena is sized
  to the cap, not to the checkpoint value.
- When `-n` exceeds `--max-seq-len`, we refuse with a clear error
  rather than silently truncating. (Upstream's behavior is silent
  truncation via `steps = config.seq_len`; the flag inverts that so the
  user can't accidentally over-request generation.)
- Default (`-1` sentinel) preserves upstream behavior — no regression
  for any existing invocation.

### Forward-path impact

Zero. `forward()` indexes the KV cache by `loff + pos * kv_dim` and
uses `p->seq_len` only as the per-layer stride. Capping `p->seq_len`
shrinks that stride but keeps the arithmetic byte-identical — the only
observable change is less memory allocated.

### Memory impact

Pre-capture: `stories42M_q80` at `--max-seq-len 256` (matching
`stories15M`'s seq_len) is expected to drop peak demand from 74.50 MB
to ~50 MB, taking 42M below the 46.55 MB physical ceiling and
eliminating all paging.

Actual measurement will be captured during task #5 integration — see
§"Integration measurements" below.

## Task #2 — Tolerance-based correctness gate

Phase 1's gate was "first 192 bytes of vellm.exe stdout byte-identical
to `tests/golden/once_upon_a_time.txt`". This works as long as the
build's floating-point semantics exactly match the golden's — which
Phase 1/Phase 2 builds achieve via `-ffloat-store` + strict x87.

Phase 3's speed work changes that. `-ffast-math`, int8 KV cache, and
tiled-matmul reassociation all perturb fp accumulation enough to shift
one or two late-sequence tokens without corrupting the model. A
byte-identical gate rejects both "optimization produced different but
still-correct output" and "optimization broke the compute path". We
need to distinguish those.

### Gate design — the simplest thing that could work

The gate is now two tries, in order, gate passes if either passes:

```
(1) PRIMARY — first 192 bytes byte-identical to golden.
              Fp-strict builds continue to satisfy this; no change.

(2) TOLERANCE — (2a) first 30 bytes byte-identical to golden AND
                (2b) ≥ 97% positional agreement on whitespace-split
                     word tokens between vellm output and golden.
```

(2a) prevents a totally-broken build from slipping through — any
meaningful tokenizer / embedding / first-layer regression corrupts
byte 1. 30 bytes ≈ 7 tokens ≈ "Once upon a time, there was", which
is inside the regime that is stable across every FP configuration
we've tested (see phase1-notes §"FP precision mismatch").

(2b) scores the body of the generation. Whitespace words are a coarser
granularity than BPE tokens (140 words vs ~200 BPE tokens in the
692-byte reference) but they're simple to compute in shell, don't
require running the tokenizer, and track semantic divergence well — a
single flipped argmax mid-stream typically perturbs 1–3 subsequent
words before the text re-aligns on the next `.` or `,`.

### Why not run_host on the fly

The gate-design tip from team-lead suggested token-agreement between
`vellm.exe` and `run_host` outputs. We didn't go that way — the
committed `tests/golden/once_upon_a_time.txt` was generated by
`run_host` at the pinned UPSTREAM_SHA and serves the same purpose
without the round-trip of re-running run_host on every gate invocation.
If the golden ever gets out of sync (e.g., upstream SHA bumps), regen
it with one line:

```bash
./run_host models/stories15M_q80.bin -z models/tokenizer.bin \
    -t 0 -s 42 -n 200 -i "Once upon a time" > tests/golden/once_upon_a_time.txt
```

### Threshold calibration

At 97% × 140 = 135.8, the required-agree count rounds up to `136`.
Empirically: a single-word perturbation gives 139/140 → passes; a
total garbage-after-first-30-bytes output gives <20/140 → fails hard.
The 5-word slack is consistent with "≥ 195/200 BPE tokens agree" from
the task brief when scaled by the 140-word/200-token ratio.

### Escalation path

If a real Phase 3 optimization emerges that produces valid output but
diverges too much at the whitespace-word granularity to clear 97%,
escalate to:

1. **BPE-token granularity.** Add a tiny host-side tool that encodes
   both vellm stdout and golden through the `tokenizer.bin` and
   positionally compares token IDs. Higher resolution (200 tokens vs
   140 words), same threshold semantics.
2. **Logit-trace diff.** Add `-DVELLM_LOGIT_TRACE` to emit the top-5
   logits per step. Cosine-similarity between vellm's logit trace and
   run_host's logit trace is the cross-toolchain-robust numerical
   comparison every serious inference project ends up with. Higher
   instrumentation cost; reserve for when coarser gates fail.

Neither is needed yet. Documenting so the next person isn't starting
from scratch if they do.

### Gate invocation

No change to caller contract. `tests/run-golden.sh` exits 0 on pass
(either gate), nonzero on fail. The stdout message differentiates
"PASS (primary)" from "PASS (tolerance)" so log scrapers can
distinguish the cases without parsing.

## Task #3 — Matmul speed optimization

Tuner's lane. Structured as experiments A/B/C/D (compiler flags →
U/V pipe pairing → blocked tiling → fixed-point matmul); each is an
atomic keep-or-discard decision logged here.

Canonical benchmark throughout: `tests/run-golden.sh` = `stories15M_q80
-t 0 -s 42 -n 200 -i "Once upon a time"` under DOSBox-X memsize=48,
cycles=fixed 90000. Iteration uses `-n 20` for sub-minute signal;
final keeper numbers are `-n 200`. Phase 2 baseline is 5m45s (commit
b5d72d0, `-O2 -fomit-frame-pointer -ffloat-store`).

### Experiment A — aggressive compiler flags

**Hypothesis:** the matmul hot loop (`src/vellm.c` `matmul()`,
specifically the Q8_0 inner `for (k=0; k<GS; k++)` integer-MAC chain
at lines 421-427) is a tight fixed-iteration loop — exactly what
`-funroll-loops` exists for. `-ffast-math` and `-O3` are additional
free-money levers if they pass gate.

**Flag sweep at `-n 20` (wall time in seconds):**

| Flags added                                                | Wall  | Speedup | Gate (n=20)  |
|------------------------------------------------------------|------:|--------:|:-------------|
| (baseline: -O2 -ffloat-store)                              | 36.80 |      —  | BYTE_ID       |
| -O3                                                        | 36.86 |  +0.2%  | BYTE_ID       |
| -funroll-loops                                             | 25.19 | -31.5%  | BYTE_ID       |
| -fschedule-insns -fschedule-insns2                         | 36.83 |   0.0%  | BYTE_ID       |
| -O3 -funroll-loops                                         | 25.11 | -31.8%  | BYTE_ID       |
| -O3 -fschedule-insns -fschedule-insns2                     | 36.93 |  +0.4%  | BYTE_ID       |
| -O3 -funroll-loops -fschedule-insns -fschedule-insns2      | 33.01 | -10.3%  | BYTE_ID       |
| -ffast-math                                                | 36.55 |  -0.7%  | BYTE_ID       |
| -funroll-loops -ffast-math                                 | 24.78 | -32.7%  | BYTE_ID       |
| -O3 -funroll-loops -ffast-math                             | 25.02 | -32.0%  | BYTE_ID       |

Reading:
- **`-funroll-loops` is the whole story.** Alone it lops 32% off wall.
- `-O3` adds nothing measurable on this workload and slightly regresses
  when `-fschedule-insns{,2}` pull in with it. Kept at `-O2`.
- `-fschedule-insns/2` is neutral or harmful (the gcc Pentium scheduler
  seems to already do most of the useful reordering at -O2; explicit
  scheduling here fights unrolling's register pressure).
- `-ffast-math` on top of unroll adds ~1.6% more at n=20. Small win on
  its own but it unlocks the big follow-up: dropping `-ffloat-store`.

**Final `-n 200` gate validation (the numbers that ship):**

| Configuration                                  | Wall    | Speedup | Gate           |
|------------------------------------------------|--------:|--------:|:---------------|
| Baseline (-O2 -ffloat-store)                   | 5m45s   |    —    | primary        |
| -funroll-loops                                 | 3m55s   | -31.9%  | primary        |
| -funroll-loops -ffast-math                     | 3m49s   | -33.6%  | primary        |
| -funroll-loops -ffast-math, drop -ffloat-store | **2m43s** | **-52.8%** (**2.11×**) | primary |
| -O3 -funroll-loops -ffast-math, drop -ffloat-store | 3m15s | -43.2% | primary        |

**Kept:** `-O2 -fomit-frame-pointer -funroll-loops -ffast-math -Wall
-Isrc` — no `-ffloat-store`, no `-O3`.

**Why dropping `-ffloat-store` is the dominant win:** `-ffloat-store`
forces x87 to spill every fp intermediate to a 4-byte memory slot and
reload it on every use, which reinstates strict IEEE-754 32-bit
rounding but costs two memory ops per fp instruction. On the P5's
short pipeline this is enormous — the FPU is blocked on the load/store
AGU every op instead of chaining. Removing it means x87 keeps 80-bit
extended precision in st(0)-st(7) registers, chains freely, and finishes
the matmul accumulation ~45% faster than the strict-rounded variant.

Phase 1 added `-ffloat-store` to guarantee byte-identity with the
SSE2-generated golden. Empirically: the 192-byte correctness
fingerprint survives the extended-precision pipeline as well — divergence
still happens somewhere past byte 192 (see phase1-notes.md §"FP
precision mismatch"), but the first 30 tokens stay on the same
cross-toolchain manifold. The gate still passes PRIMARY at n=200, which
is as strong as Phase 1's result.

**Why -O3 regresses vs -O2 here:** -O3 enables `-ftree-loop-vectorize`
and `-fpredictive-commoning`, neither of which help on a Pentium
without SSE. They enlarge the code footprint (the P5 has only 8 KB
I-cache) and add register pressure that hurts the now-unrolled matmul.
`-O2 -funroll-loops` is the sweet spot for P54C.

**Decision:** kept. Commit: phase3-task3-flags.

**Speedup vs Phase 2 baseline after Experiment A:** 5m45s → 2m43s
= **2.11× faster, -52.8% wall time**. Already past the "measurable
speedup" exit bar for Phase 3 in a single commit.
